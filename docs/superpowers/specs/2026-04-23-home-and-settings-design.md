# Inkbloom 首页与设置设计文档

日期：2026-04-23

## 1. 目标

本次设计在现有 MVP 工作台基础上补齐一个真正可进入应用的启动首页，并增加最小可用的全局设置页。

这次改动只解决两个问题：

- 应用启动后先进入一个明确的首页，而不是直接掉进创作工作台
- 用户可以在应用内配置全局 LLM 接口信息

这次不扩展更多设置项，也不改写现有创作工作台的核心结构。

## 2. 范围

本次设计范围固定为以下两部分：

- 首页书籍管理：创建、修改、删除、进入创作
- 设置页：全局 LLM 配置

明确不在本次范围内的内容：

- 项目级 LLM 配置
- 连接测试、模型拉取、额度检查
- 复杂的书籍元数据管理
- 首页搜索、筛选、排序、分页
- 多用户、多工作区或云同步

## 3. 信息架构

应用拆分为三个顶层视图状态：

- `Home`：启动首页，承担书籍管理和设置入口
- `Settings`：全局设置页，目前只负责 LLM 配置
- `Workspace`：进入某本书之后的创作工作台

基础流转如下：

1. 应用启动后默认进入 `Home`
2. 用户在 `Home` 选择一本书后进入对应 `Workspace`
3. 用户在 `Home` 点击新建，创建成功后直接进入新书的 `Workspace`
4. 用户在 `Home` 点击设置后进入 `Settings`
5. 用户在 `Workspace` 中可以返回 `Home`

这一版不引入正式路由系统，直接由顶层 React 状态控制视图切换即可。原因很简单：当前页面总数很少，引入路由只会增加状态同步复杂度。

## 4. 首页设计

### 4.1 首页定位

首页是应用启动后的唯一入口页，语义上相当于书架或启动台。用户先决定“要进入哪本书”，然后才进入创作工作台。

这意味着首页不属于当前工作台的一部分，而是工作台之外的一层壳。

### 4.2 首页布局

首页采用书架卡片式方案。

页面由以下区域组成：

- 顶部标题区：展示 `Inkbloom` 名称和简短说明
- 主内容区：展示书籍卡片列表
- 顶部或右上角次级操作区：提供 `新建书籍` 和 `设置`

书架卡片式优于列表表格和详情面板式的原因：

- 更符合“选一本书继续创作”的使用心智
- 与写作工具的气质更一致，不像后台管理页面
- 对当前 MVP 更轻，不会过早进入复杂项目管理形态

### 4.3 书籍卡片

每张书籍卡片至少展示：

- 书名
- 最近更新时间
- 主动作：`进入创作`
- 次动作：`修改`
- 次动作：`删除`

主动作必须和管理动作视觉上区分开，避免用户误触删除或修改。

### 4.4 空状态

当用户还没有任何书籍时，首页显示明确空状态。

空状态只强调两个动作：

- `新建第一本书`
- `去设置 LLM`

这样能避免空页面给用户造成“应用还不能用”的错觉，同时保持首次体验足够收敛。

## 5. 书籍管理交互

### 5.1 创建书籍

创建书籍通过首页弹窗完成，而不是单独进入一个书籍编辑页。

原因如下：

- 当前书籍对象很轻，主要只有书名等最小元数据
- 为轻量对象单独开页会让交互层级变重
- 启动首页天然适合承接这种短流程操作

创建流程如下：

1. 用户点击 `新建书籍`
2. 打开 `BookDialog`
3. 用户填写书名并确认
4. 主进程创建书籍记录和对应本地目录结构
5. 创建成功后直接进入该书的 `Workspace`

这一步已经确定不让用户创建后留在首页，因为直接进入更符合启动台心智，也减少了一次额外点击。

### 5.2 修改书籍

修改书籍同样复用 `BookDialog`。

流程如下：

1. 用户点击某张卡片上的 `修改`
2. 弹窗回填当前书名
3. 用户修改并保存
4. 首页卡片即时刷新

这一版不新增复杂字段，因此“创建”和“修改”可以共享同一套表单结构。

### 5.3 删除书籍

删除书籍通过单独的确认弹窗完成，不能直接一击删除。

确认文案必须明确表达这会删除本地书籍项目，而不是仅仅从首页移除。

流程如下：

1. 用户点击卡片上的 `删除`
2. 打开二次确认弹窗
3. 用户确认后执行删除
4. 删除成功后刷新首页列表

如果删除失败，用户停留在确认态并看到错误信息，避免静默失败。

## 6. 设置页设计

### 6.1 设置页定位

设置页是独立页面，不和首页混排，也不塞进当前工作台。

原因是这类配置行为和“选书进入创作”是两种不同任务。独立成页后，信息边界更清楚。

### 6.2 本次设置项

本次只支持全局 LLM 配置，字段严格收敛为：

- `provider`
- `baseUrl`
- `apiKey`
- `model`

明确不加入以下字段：

- `temperature`
- `maxTokens`
- `timeout`
- `topP`
- 自定义请求头

这样做的原因不是这些字段不重要，而是当前 MVP 还没有足够多的 provider 差异和调优需求。过早暴露大量参数，只会把“设置页”做成一个不必要的配置中心。

### 6.3 保存行为

设置页支持查看、编辑、保存和返回。

保存成功后保留在当前页，由用户自行点击返回首页，而不是保存后自动跳转。这样更稳，也更符合“设置完成后再决定下一步”的预期。

如果保存失败，表单内容必须原地保留，方便用户修改后重试。

## 7. 数据边界

### 7.1 书籍记录

应用级书单需要有一个最小记录结构，至少包含：

- `id`
- `title`
- `updatedAt`

后续如果需要增加简介、封面、路径等信息，可以继续扩展，但这次设计不以这些字段为前提。

### 7.2 当前激活书籍

当前进入哪一本书，只作为应用顶层状态保存在前端，用来决定是否展示 `Workspace`。

### 7.3 全局设置

LLM 设置属于应用级配置，不和任意一本书绑定。

这意味着：

- 用户换书时不需要重新配置模型
- 新建书籍默认继承当前全局设置能力
- 后续如果要扩展项目级覆盖配置，也可以在此基础上加一层，而不是推翻现有结构

## 8. 错误处理

### 8.1 未配置 LLM

如果用户尚未配置 LLM：

- 不阻止用户创建书籍
- 不阻止用户进入工作台
- 但首页或工作台应明确提示 AI 能力当前不可用

这样能保证“本地优先”原则不被破坏。LLM 只是能力增强，不应成为打开和管理作品的前置条件。

### 8.2 书籍创建失败

如果书籍目录创建失败，则不能保留半成品书籍记录。要么完整成功，要么完整失败，避免首页出现打不开的脏数据。

### 8.3 设置保存失败

设置保存失败时保留用户输入，不清空表单，不自动返回。

### 8.4 记录存在但目录缺失

如果某本书记录还在，但底层目录已经缺失，则首页卡片应显示异常状态，并阻止直接进入工作台。

这一版可以先只做错误提示，不急着补“自动修复目录”能力。

## 9. 对现有 MVP 的影响

这次改动不会重做当前三栏工作台，而是在它外面补一层应用壳。

主要影响点如下：

- `App` 顶层要从“直接渲染工作台”改成“根据视图状态渲染 Home / Settings / Workspace”
- `Workspace` 内需要一个返回首页的稳定入口
- 应用需要新增书籍列表和全局设置的读取与保存接口

从实现顺序看，应先完成顶层状态切换和首页，再补设置页，最后接通现有工作台的进入与返回流转。

## 10. 结论

本次设计采用以下最终方案：

- 启动后进入独立首页
- 首页采用书架卡片式布局
- 书籍创建与修改都用轻量弹窗完成
- 删除使用二次确认弹窗
- 新建成功后直接进入创作工作台
- 设置页独立成页，且只支持全局 LLM 配置

这个方案在当前 MVP 阶段最稳。它把入口层、管理层和创作层明确分开，同时保持实现复杂度可控，不会因为过早设计复杂项目管理或高级配置中心而分散精力。
