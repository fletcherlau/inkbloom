# Inkbloom 设计文档

日期：2026-04-22

## 1. 产品定义

Inkbloom 是一款本地优先、以对话为主入口的长篇小说创作工作台。

它的目标不是提供一个“接了大模型的写作编辑器”，而是帮助作者在长篇创作中持续管理作品状态、降低冷启动难度、减少设定漂移，并通过结构化工作流让模型在正确的阶段做正确的事。

第一版产品定位如下：

- 运行形式：本地桌面应用，核心界面使用 Web 技术实现
- 主要交互：Chat-first，文档查看和轻编辑为辅，改写和 Agent 任务为次级能力
- 模型策略：以云端大模型为主
- 创作对象：长篇小说项目，而非单篇零散文本
- 系统能力：不仅能回答问题，还能根据项目当前阶段和可用 skill 推进创作流程

## 2. 设计原则

### 2.1 本地优先

作品的核心内容、结构状态和工作流进度都保存在本地。即使没有网络，用户仍然可以打开项目、阅读、编辑、整理结构。联网仅影响模型能力，不影响作品本身的可用性。

### 2.2 Chat-first

用户最常用的入口应当是对话，而不是复杂编辑面板。系统应尽量在聊天中完成整理、发散、收敛、生成、检查和推进，只在必要时要求用户进入专门页面进行查看、确认或精修。

### 2.3 结构化创作

系统不以聊天记录作为唯一上下文，而是将作品拆分为长期知识层和正文层。所有模型调用都应基于结构化上下文选择，而不是临时拼接整本小说内容。

### 2.4 渐进式引导

系统应能判断当前创作阶段、识别缺口，并在适当时机建议下一步。用户可以随时跳步、打断或改方向，但默认体验应鼓励持续迭代。

### 2.5 可回写、可追踪

AI 产出不应只存在于聊天消息中。需要区分哪些结论已经被确认、写入了 Story Bible，哪些内容只是临时草稿、候选文本或待执行任务。

## 3. 核心对象模型

### 3.1 Story Bible

Story Bible 是作品级知识底座，负责保存“这本书是什么”，用于维持长期一致性和高质量上下文调用。

第一版采用以下分类：

- Braindump：灵感池，容纳未整理碎片、场景、片段、对白、想法
- Genre：类型和体裁约束，定义读者预期和叙事惯例
- Style：文风、叙事视角、节奏偏好、禁忌写法
- Synopsis：故事梗概，包括一句话梗概、一段式梗概和扩展梗概
- Characters：角色档案、关系、动机、弧光、显著特征
- Worldbuilding：世界观、规则、制度、历史、地理、科技或魔法体系
- Outline：卷、幕、章级结构和关键情节点
- Themes：主题与母题，用于保持长篇作品的精神内核
- Constraints：硬性创作约束，用于在模型生成和检查时强绑定

### 3.2 Manuscript

Manuscript 是正文层，负责保存“这本书现在写到了哪”。它包含卷、章节、场景和历史版本，是用户实际写作和修订的主承载层。

### 3.3 Task

Task 用于承载需要多步骤处理的复杂动作，例如从灵感池提炼大纲、扫描章节一致性、汇总前文并更新梗概等。Task 是 Agent 式流程的外显承载体，必须有状态和中间结果。

### 3.4 Workflow State

Workflow State 保存当前作品所处阶段、阶段完成度、系统建议的下一步动作，以及相关 skill 的执行记录。它是系统“会推进”而不是“只回答”的关键。

## 4. 运行架构

### 4.1 技术选型

第一版推荐技术栈：

- 桌面容器：Electron
- 前端：React + TypeScript + Vite
- UI：shadcn/ui + Tailwind CSS
- 富文本编辑：Tiptap
- 本地状态管理：Zustand
- 数据校验：Zod
- 本地数据库：SQLite
- 模型编排层：Node.js + TypeScript
- 本地全文检索：SQLite FTS

选择理由：

- Electron 对本地项目目录、文件读写、系统能力和快速试错最友好
- React + TypeScript 适合承载复杂状态联动
- Tiptap 比简单文本域更适合未来承载结构化标注和候选改写结果
- SQLite 足以支撑单机单用户的本地优先场景
- Node/TypeScript 适合快速迭代多模型接入、上下文编排和任务流逻辑

### 4.2 进程结构

应用内部拆为三层：

- Renderer：React 前端，负责界面和交互
- Main：Electron 主进程，负责窗口、文件系统、系统集成
- Local App Service：本地服务层，负责数据库访问、上下文编排、模型调用、任务调度

前端不直接访问外部模型 API，也不直接承担工作流编排职责。所有模型调用和上下文选择统一由本地服务层处理。

### 4.3 项目目录

每部作品对应一个本地目录，建议结构如下：

```text
MyNovel/
  manuscript/
    volume-01/
      chapter-001.md
      chapter-002.md
  assets/
    characters/
    locations/
    references/
  exports/
  snapshots/
  .inkbloom/
    project.db
    settings.json
    cache/
```

说明：

- `manuscript/` 保存正文章节和卷结构
- `assets/` 保存参考素材和附件
- `exports/` 保存导出稿
- `snapshots/` 保存关键版本快照
- `.inkbloom/project.db` 保存结构化数据、索引、任务和缓存

### 4.4 文件与数据库边界

适合存文件：

- 正文和章节
- 导出稿
- 附件素材
- 快照副本

适合存数据库：

- 章节索引
- Story Bible 条目
- 角色关系
- 时间线与伏笔映射
- 摘要缓存
- 任务执行记录
- 成本统计和模型调用日志
- 当前工作流阶段和完成度

原则是：创作原文尽量文件化，系统状态尽量数据库化。

## 5. AI 交互模型

### 5.1 主入口：Chat

第一版的主入口是对话。用户应尽量在聊天中完成绝大多数创作动作，包括：

- 发散灵感
- 整理想法
- 提炼梗概
- 构建角色
- 推进大纲
- 续写章节
- 检查冲突
- 归档到 Story Bible
- 触发复杂任务

这意味着产品体验应更像“懂这本书的创作搭子”，而不是要求用户先学习一套复杂编辑软件。

### 5.2 次级入口：文档查看与轻编辑

用户可以查看并轻度编辑 Story Bible 条目、章节内容和任务结果，但这些页面主要用于确认、修正和补充系统产出，而不是要求用户主动维护复杂系统。

### 5.3 次级能力：Rewrite

改写能力不是第一版的主体验，只在下列场景中突出：

- 某段已有内容需要润色
- 某章需要扩写或缩写
- 某段需要改视角或改语气
- 需要对比两个版本并选择其一

系统应以候选补丁或候选版本形式返回结果，不直接覆盖用户正文。

### 5.4 高级能力：Agent Task

复杂、多步骤、跨度较大的请求进入 Task 工作流。例如：

- 将 Braindump 整理成可用大纲
- 汇总前十章并更新故事梗概
- 扫描角色出场并生成一致性报告
- 根据现有大纲扩写某一卷的章节草案

Task 需要展示状态、中间结果和用户确认点，避免黑盒化。

## 6. 上下文编排

长篇创作的核心不在于接入模型，而在于每次调用时带上正确的上下文。

第一版将上下文分为五层：

- 当前操作对象：当前章节、当前选中文本、当前 Bible 条目
- 邻近结构：上一章、下一章、本卷大纲
- 长期设定：人物卡、世界观、风格、约束
- 历史摘要：更早正文的压缩摘要
- 显式钉住内容：用户本次明确指定必须参考的材料

SudoWrite 式的 Story Bible 分类不只是 UI 组织方式，同时还是上下文编排的输入层。例如：

- 写人物对话时优先使用 Characters + Style + 当前章节
- 扩写章节时优先使用 Outline + Synopsis + 邻近章节
- 检查设定冲突时优先使用 Worldbuilding + Constraints + 相关角色
- 从灵感池提炼结构时优先使用 Braindump + Themes + Genre

## 7. 工作流状态机

第一版不做完全自由的通用 Agent，而是做“有状态的本地创作工作流”。

高层阶段定义如下：

- Ideation：灵感发散和 Braindump 收集
- Foundation：建立 Story Bible 基础内容
- Outline：把故事推进到卷、幕、章结构
- Drafting：开始实际章节写作
- Revision：进行一致性修订和精修
- Export：整理成可交付稿件

系统判断当前步骤时采用三层机制：

1. 显式项目状态：数据库中保存当前阶段
2. 结构完成度：基于已有梗概、角色、大纲、章节等判断缺口
3. LLM 判断：在状态和完成度基础上判断下一步最合理动作

这种设计比纯自由 Agent 更稳，也比纯规则系统更灵活。

## 8. Skill 机制

Skill 是工作流节点，而不是任意 prompt 工具集合。

第一版建议实现以下核心 skill：

- `capture_braindump`：将聊天中的碎片灵感保存为 Braindump
- `shape_synopsis`：将零散想法提炼为一句话、短梗概、长梗概
- `build_characters`：提取和扩展角色卡
- `build_world`：整理世界观规则和设定边界
- `build_outline`：将梗概推进为卷、幕、章结构
- `draft_scene`：基于当前上下文生成场景或章节草稿
- `consistency_check`：检查角色、时间线、设定、风格冲突
- `revision_pass`：执行目标明确的修订任务
- `update_bible`：将对话中确认的信息回写到 Story Bible
- `next_step_guide`：判断下一步最合理动作，并引导用户持续推进

其中 `next_step_guide` 是第一版最重要的引导 skill。它负责让系统从“被动回答”提升为“主动推进”。

## 9. 雪花写作法的产品化

雪花写作法适合作为第一版的启动工作流模板。它不应只是文档说明，而应直接成为新建作品时的可选初始化路径。

新建作品时提供三个入口：

- 空白作品
- 雪花写作法启动
- 从已有稿件导入

雪花写作法各步的结果应自动落入 Story Bible 和 Manuscript，而不是停留在聊天记录中。建议映射如下：

- 一句话故事核心 -> Synopsis
- 一段式梗概 -> Synopsis
- 角色摘要 -> Characters
- 扩展剧情段落 -> Outline
- 人物深描 -> Characters
- 场景列表 -> Outline 或 Braindump
- 章节映射 -> Outline
- 试写片段 -> Manuscript 或 Braindump

每一步都应支持两种操作：

- 我自己写
- 让 AI 帮我生成初稿

## 10. 界面信息架构

第一版的主界面应以聊天为中心，而不是以编辑器为中心。建议布局如下：

- 左侧：作品结构导航，包含 Story Bible、Manuscript、Tasks
- 中间：主对话区，作为默认焦点
- 右侧：上下文面板，显示当前对话已加载的内容、推荐引用和建议动作

其他页面包括：

- 项目首页：显示作品概况、最近章节、最近对话、待办任务
- Story Bible 页：浏览和轻编辑各分类条目
- 章节页：查看和撰写章节，支持续写和候选改写
- 任务页：查看任务状态、中间结果、确认点和历史记录

用户体验重点在于让用户理解三件事：

- 当前正在处理什么对象
- AI 当前参考了哪些上下文
- 这次产出将写回到哪里

## 11. 第一版 MVP 范围

第一版必须证明的核心价值是：

它是否能够让作者稳定地写一部长篇，并且比“普通聊天工具 + 文档文件夹”更不容易失控。

建议第一版只做以下能力：

- 本地项目创建、打开、保存
- Story Bible 基础分类和条目管理
- Chat 主入口
- 带上下文的对话能力
- 将对话结果保存到 Story Bible 或章节
- 基础工作流状态和下一步建议
- 首批核心 skill
- 章节查看和基础写作
- 轻量一致性检查

第一版明确不做：

- 多人协作
- 云同步
- 账户系统
- 插件市场
- 自动生成整本小说
- 通用智能体平台
- 复杂地图或高度可视化世界构建工具
- 私有微调训练

## 12. 第一版数据模型

建议以最小可行模型启动：

- `Project`
- `BibleItem`
- `Chapter`
- `ChatThread`
- `ChatMessage`
- `Task`
- `Snapshot`
- `ContextPreset`
- `WorkflowState`

其中 `BibleItem` 使用 `type` 区分 Braindump、Character、Outline、Style 等具体类别。

## 13. 第一版开发顺序

### 13.1 基础骨架

- 建立 Electron + React + TypeScript 项目结构
- 建立本地项目目录与 SQLite 存储
- 完成项目创建、打开、保存

### 13.2 Story Bible 基础层

- 支持 Braindump、Synopsis、Characters、Worldbuilding、Outline、Style、Genre
- 完成基础查看、创建、更新

### 13.3 Chat 主入口

- 支持基础对话
- 支持显式上下文选择
- 支持将结果保存到 BibleItem 或 Chapter

### 13.4 工作流与引导

- 建立 Ideation、Foundation、Outline、Drafting 四阶段
- 建立完成度判断和下一步建议

### 13.5 首批 skill

- `capture_braindump`
- `shape_synopsis`
- `build_characters`
- `build_outline`
- `next_step_guide`

### 13.6 章节写作与检查

- 章节页基础写作
- 章节续写
- 轻量角色/设定/大纲偏离检查

## 14. 风险与设计取舍

### 14.1 过早做重编辑器

如果过早把产品做成复杂编辑器，会提高学习成本，偏离 Chat-first 原则。第一版应克制高级排版和复杂局部操作，保持对话体验为核心。

### 14.2 过早做通用 Agent

如果过早追求全自动创作 Agent，系统会变得不稳定且不可预测。第一版应以有限状态和有限 skill 为主，强调可追踪和可回写。

### 14.3 过早引入云端复杂度

账户系统、同步和协作会显著增加复杂度，但不直接帮助验证单人长篇创作价值。第一版应严格保持单机单用户边界。

### 14.4 过早引入复杂检索栈

在项目规模尚可控时，没有必要引入独立向量数据库。第一版优先使用 Story Bible 分类、结构化关系、SQLite FTS 和摘要缓存即可。

## 15. 成功标准

第一版完成后，应至少满足以下标准：

- 用户可以本地创建并持续使用一个长篇项目
- 用户可以主要通过聊天推进创作，而非依赖复杂面板
- 系统可以把对话中的结论结构化写入 Story Bible
- 系统可以基于当前阶段给出合理下一步建议
- 系统可以在章节写作时引用长期设定并提示明显冲突
- 用户能明确知道 AI 参考了什么，以及结果被保存到哪里

达到这些标准，才说明 Inkbloom 已经具备区别于通用聊天工具的真实产品雏形。
